What the model predicts

The model predicts how much CPU time a process is likely to need. The scheduler then turns that prediction into a score. A lower score means “run this process sooner.”

score = predicted_runtime + 0.65 x deadline_slack - 2 x priority

The running app loads an XGBoost model trained on a normalized sample of the real Alibaba cluster trace. When you enter tasks manually, the frontend sends every task to the backend, the backend predicts all runtimes in one batch, then the scheduler uses those predictions.

io_durationExpected I/O wait duration after blocking.

Training flow

  1. Download Alibaba `batch_task.csv`.
  2. Normalize rows into simulator tasks.
  3. Export runtime features with `--mode export-features`.
  4. Train XGBoost to predict observed runtime.
  5. Load the trained model in the backend.
  6. Batch-predict manual or generated tasks before scheduling.

Why this is useful

FIFO is easy to reason about but ignores urgency. MLFQ responds quickly but can create high context-switch overhead. The ML-guided policy tries to identify short, urgent, high-priority work before it turns into deadline misses.
