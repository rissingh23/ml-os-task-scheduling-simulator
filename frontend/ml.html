What the model predicts

The online scheduler estimates how expensive a ready task is likely to be, then combines that runtime estimate with deadline pressure and priority. Lower score wins.

score = predicted_runtime + 0.65 x deadline_slack - 2 x priority

The manual page uses a bundled linear model trained from simulator-exported features. The backend pipeline also supports XGBoost: Python trains the model and writes batched task predictions, while the C++ scheduler consumes those predictions with cheap lookup.

XGBoost deployment

Use `batch_predict_xgboost.py` to write `task_id,predicted_runtime`. This avoids running tree inference inside the scheduler loop.

Training status

Yes: `scripts/train_linear.py` trains the linear runtime predictor consumed by C++ and mirrored in `frontend/trained_model.js` for the manual page.

What it learned

With the current synthetic labels, runtime is mostly explained by burst length. The UI shows each feature contribution so this is visible instead of hidden.

Features

Exported by the simulator
requested_runtimeTask runtime request or remaining CPU burst estimate.
deadline_slackTime left before deadline at the dispatch point.
priorityUser or workload priority, higher means more urgent.
io_intervalHow frequently the task blocks for I/O.
io_durationExpected I/O wait duration after blocking.
ran_since_ioCPU time consumed since the last I/O block.

Training flow

  1. Generate or normalize task traces.
  2. Export runtime features with `--mode export-features`.
  3. Train a linear model or XGBoost regressor.
  4. Load the linear model into the C++ online scheduler.
  5. Mirror the trained linear weights in the browser for inspectable manual scheduling.

Why this is useful

FIFO is easy to reason about but ignores urgency. MLFQ responds quickly but can create high context-switch overhead. The ML-guided policy tries to identify short, urgent, high-priority work before it turns into deadline misses.
